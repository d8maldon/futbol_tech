## Critical (high)

### 1. match_data.py win-prob/score-timeline core is entirely untested yet runs fully offline
- **Where:** src/match_data.py - load() score timeline lines 117-127 and 132-144; wp() lines 64-72; calibrate() lines 50-61  _(regression/test-coverage)_
- **Issue:** This module computes the dashboard's headline numbers: the running score from goal events (with the documented FotMob quirk that homeScore/awayScore is the score BEFORE the goal, so it COUNTS goals — a fragile invariant), the calibrate() grid-search that anchors minute-0 win-prob to the pre-match model, and the wp() Skellam remaining-goals curve. I verified all three run with no data/ dependency: MD.wp/MD.calibrate execute directly, and wp() sums to 1.0 across a (d in -2..2)x(t in 0,10,45,80,90,100) grid (checked: zero violations), with correct terminal behavior (t>=90 collapses to a degenerate {win/draw/loss}=1). None of this is asserted anywhere, so a sign flip in `wp(hs - as_, ...)`, an off-by-one in the `goals[gi]["min"] <= t` advance, or a mis-anchored calibration would ship silently.
- **Fix:** Add tests/test_match_data.py (no data needed): (1) wp() — assert sum==1 over a grid, monotonicity (P(home win) increases in d), and the t>=90 degenerate cases; (2) calibrate() — assert the returned (mh,ma) reproduces the input p_h/p_d within the grid step (skellam.sf(0,mh,ma)~=p_h); (3) build a tiny synthetic match dict and assert the score timeline counts goals correctly and that wp_home[0] ~= pre_match p_h (the anchoring claim in the docstring).

### 2. 12 m match gate discards mis-placed players, inflating the 'within 5 m' and mean-error headline
- **Where:** src/validate_topdown.py - match_err lines 110-117; validate_clip lines 154-157; main lines 184-186  _(accuracy/cv)_
- **Issue:** match_err runs linear_sum_assignment(our->GT) then returns only e[e <= gate] with gate=12.0 (line 117). Every player our pipeline placed more than 12 m from its assigned ground-truth player is dropped from the error array entirely rather than counted as a large error or a false positive. The pooled mean, median, 'within 5 m', and 'within 10 m' figures in main (lines 184-186) and the README headline ('Mean localisation error 5.1 m ... 53% within 5 m', README.md:208-209) are therefore computed only over the subset that already matched well. Gross homography failures, duplicate detections, and crowd/bench false positives never enter the statistic, so the reported accuracy is an optimistic lower bound on true error, not the unconditional localisation error a reader assumes.
- **Fix:** Report the gate's effect explicitly: count unmatched/over-gate detections as misses (e.g. precision/recall or a capped error), or report the fraction of our detections that fall within 5 m of any GT alongside the matched-only mean. At minimum, state in README/HANDOFF that the 5.1 m / 53% figures are conditioned on a 12 m matching gate.

### 3. Per-clip 4-way x/y flip is selected against ground truth, partially fitting the test set
- **Where:** src/validate_topdown.py - match_err line 113 (o = ours * flip); validate_clip lines 146-153 (flips, flip_score, bf = max(...))  _(accuracy/cv)_
- **Issue:** validate_clip enumerates flips [(1,1),(-1,1),(1,-1),(-1,-1)] and picks bf = the flip maximising matches then minimising total error against GT (flip_score, lines 148-153), applying it via o = ours*flip in match_err. The 32-keypoint pitch model emits IDENTIFIED landmarks (corner (0,0) vs (105,0), etc.), so a correct homography already fixes orientation; a winning (-1,1) or (1,-1) means the model/GT disagree on left-right or near-far identity, and resolving that by choosing the flip that best matches GT is a per-clip degree of freedom fit to the labels. This both hides a real keypoint-identity/mirror failure mode and makes the reported error optimistic, since the orientation a live deployment cannot peek at is here chosen with the answer key.
- **Fix:** Resolve orientation from the keypoint identities / camera-side prior BEFORE seeing GT, or fix a single global convention and report error without per-clip GT-driven flip selection. If a flip is genuinely needed, disclose it and measure how often each flip wins (frequent non-identity flips indicate a keypoint-labeling bug worth fixing upstream).

## Medium - Accuracy

### 7. Home-field edge baked into the logit intercept is misapplied to neutral-venue WC fixtures (spurious ~9pp / ~80-Elo bias to the listed 'home' team)
- **Where:** src/ratings.py - seed_history L196-197, fit_prematch_logit L228-244, prematch_proba L255-261; applied in backtest.py dr_of L64-66 and ratings.main L315-318  _(accuracy/predictor)_
- **Issue:** The prematch logit is fit on one pooled dr where dr = (rh + ha) - ra with ha=HOME_ADV(65) for non-neutral matches and 0 for neutral. In the martj42 train window (2005-2020) ~72% of rows are non-neutral (I checked: train neutral fraction 0.278), so the model's intercept absorbs the average home edge of real home games. At dr=0 the loaded production model returns H=0.404, D=0.285, A=0.311 — a 0.093 asymmetry favoring 'home', equivalent to ~80 Elo of phantom advantage. But every WC2026 group game is at a neutral US/Canada/Mexico venue, and fixtures.py has no neutral concept — 'home'/'away' is just FIFA listing order. dr_of() only adds HOME_ADV for the three host nations, so for the vast majority (non-host) WC fixtures no ha is added, yet the logit STILL hands the nominal home team that ~9pp edge purely from listing order. The OOS 0.86/ECE-0.018 numbers don't catch this because the test set is also ~73% non-neutral, so calibration holds on that blend — but it does not hold for a genuinely neutral fixture.
- **Fix:** Either (a) add a neutral/non-neutral indicator feature to the prematch logit so the intercept doesn't conflate venue with team strength, or (b) fit the draw model only on neutral matches (or center ha so dr=0 means a true coin-flip), or (c) at minimum add +HOME_ADV symmetrically for the listed home team of every non-host WC fixture only if you intend to model a home edge — but at a neutral venue there is none, so the listing-order bias should be removed.

### 8. Champion Monte-Carlo does not use the validated draw model; README claims it is 'the very engine validated out-of-sample'
- **Where:** src/montecarlo.py - SUPREMACY L44, GOAL_TOTAL L43, sample_score L81-85, sim_knockout L195-198  _(accuracy/predictor)_
- **Issue:** montecarlo.py never imports prematch_proba/davidson_proba/the logit. Group scorelines come from a hardcoded linear map sup = dr/400*SUPREMACY (SUPREMACY=1.3) into independent Poissons with GOAL_TOTAL=2.6, and knockouts pick the winner with raw expected(dr). Neither SUPREMACY nor GOAL_TOTAL is fit or validated anywhere in the repo (grep confirms they are literals), and the calibrated 0.86-log-loss logit/Davidson model that WAS validated is entirely unused in the simulation. README L148-160 states the champion sim uses 'the very engine validated out-of-sample below' and Poisson 'drawn from the rating gap' — the title odds therefore rest on an unvalidated supremacy/Poisson parameterization, not the validated probabilities. Independent home/away Poissons also ignore score correlation (no Dixon-Coles low-score adjustment), which biases the simulated draw rate and thus GD/GF tiebreakers.
- **Fix:** Either derive the group/knockout match probabilities from the validated prematch_proba (e.g., calibrate the Poisson supremacy so its implied 1X2 matches the logit, or sample outcomes directly from prematch_proba and only sample scorelines conditional on outcome), or soften the README claim to state the simulation uses a separate, unvalidated Poisson supremacy model. Consider a Dixon-Coles correction for low-score correlation.

### 9. Headline 'subs migrate ~1.8x' is partly circular: substitution stoppages are classified AS breaks
- **Where:** src/analyze.py - classify(), lines 31-42 (drinks_break returns at line 38, before substitution at line 41)  _(accuracy/thesis)_
- **Issue:** In classify(), the drinks_break branch (gap>=90s, half_min 25-31, period<=2) is evaluated BEFORE the substitution branch (subs_in>0). So any substitution that produces a >=90s dead-ball gap in minutes 25-31 is labeled a 'drinks_break', not a 'substitution'. The break set is therefore selected partly on the presence of a long stoppage, which substitutions cause, while the control set is explicitly placed in halves with NO long stoppage (the 'blocked' filter at analyze.py:146 removes any half with a gap>=90s). The survivor finding 'subs migrate into the pause ~0.53 vs 0.29, ~1.8x, CI +0.04..+0.46' (README 'What survives') is thus measuring a quantity that the sampling design is biased toward. The README discloses this in a single parenthetical clause (lines 136-137: 'control halves cannot contain long sub stoppages by construction, and some detected breaks are themselves substitution stoppages, so part of this gap is selection'), but the figure/headline present it as the primary positive result without quantifying how much of the 1.8x is artifact.
- **Fix:** Quantify the contamination: report subs_in for detected breaks that are NOT sub-stoppages, or re-detect breaks excluding gaps whose only non-play content is a Substitution, and re-estimate the ratio. At minimum, surface the caveat at the same prominence as the 1.8x claim rather than as a trailing parenthetical.

### 10. Penalties excluded from the xT model but INCLUDED in the momentum/threat-share analysis
- **Where:** src/analyze.py - lines 56-57 (shots filtered only period<=4, NOT pen==0) and 89-94 (penalty xG added to threat); same in make_figures.py:250, 265-268  _(accuracy/thesis)_
- **Issue:** xt_model.py:29 deliberately drops penalties (pen==0) when learning the surface, with the comment that penalty-spot value 'says nothing about the value of reaching that cell in open play'. But analyze.py builds the threat series by filtering shots only on period<=4 (line 57) and then adds raw shots.xg (line 92) with no pen==0 filter, so a penalty (xG ~0.76) is dumped into the per-team threat-share and swing computation. make_figures.py does the same for the hero momentum river (line 250, 266-268). A single penalty near a break dominates the pre/post share and can manufacture or mask a 'flip'. This is both an internal inconsistency (penalties matter for momentum but not for value) and a power/bias problem for the 'momentum null' (flips 34% vs 36%, CI -14..+10): the null is the headline, and adding high-variance penalty mass to the share makes the test noisier and the comparison harder to interpret.
- **Fix:** Filter shots to pen==0 (and ideally open-play only) in analyze.py and make_figures.py threat construction, consistent with the xT model, then re-run the flip/swing CIs. If penalties are intentionally kept as 'threat that happened', document that choice and note it diverges from the xT definition.

### 11. 'Open play only' overstates the filtering: free-kick and corner shots/passes are retained
- **Where:** src/xt_model.py - line 27-29 comment 'open play only' vs shots filter (pen==0 only); moves include all passes/carries (extract.py:103-118)  _(accuracy/thesis)_
- **Issue:** The comment at xt_model.py:27 says 'open play only' and excludes only shots where shot.type == 'Penalty' (plus period>4 shootouts). StatsBomb shot types also include 'Free Kick', 'Corner', and 'Kick Off', none of which are filtered, so direct free-kick and corner shots still feed p_shot/p_goal, inflating near-goal value in ways that are not open-play. Likewise the move set (extract.py Pass/Carry) includes corner deliveries and free-kick passes, inflating transitions into the box. The README ('failed moves absorb to zero; shootout and penalty kicks excluded from the shot model', line 258; fig 'P(goal within possession from here)') and event_stats.py:13 ('open-play') all inherit this overclaim. The reported '13.6k open-play shots' / '931k passes' counts therefore are not strictly open play.
- **Fix:** Either filter to shot.type=='Open Play' and the open-play pass/carry subset to match the 'open play only' claim, or relax the wording to 'penalties and shootout kicks excluded' everywhere (README, xt_model.py comment, event_stats.py, make_figures titles).

### 12. WC 2022 is called the 'noise floor' but is not the minimum firing rate; floor inherits the same sub-stoppage contamination
- **Where:** README.md - detector table lines 26-36; floor logic analyze.py:64-77  _(accuracy/thesis)_
- **Issue:** The README table lists WC 2022 at 0.11/match as 'the measured noise floor', yet Euro 2020 (0.12) fires HIGHER and WC 2018 (0.09), WWC 2023 (0.08) fire LOWER. So WC 2022 is not an empirical floor; it is a chosen air-conditioned reference. Purity (analyze.py:77) subtracts wc22_fp_rate * n_hot_matches as 'expected background', i.e. assumes the hot tournaments' false-positive rate equals Qatar's. But the false-positive mechanism includes substitution-caused stoppages and VAR checks whose base rate plausibly differs by era/competition (WC2022 had aggressive VAR; ISL/AFCON differ), so the single-floor subtraction is a strong, unvalidated assumption. Because sub-stoppages count as detections in BOTH the floor and the break set (see finding 1), the purity subtraction partially — but not fully — nets them out only if the per-match sub-stoppage rate is equal across tournaments, which is not shown.
- **Fix:** Either justify WC 2022 as the floor explicitly (lowest among genuinely cooling-break-free hot-free tournaments) or use a pooled cool-tournament background. Sensitivity-test purity against Euro 2020 / WWC 2023 as alternative floors and report the range.

### 13. Headline '~5 m / 53% within 5 m' omits the gating and flip-fit caveats that produced it
- **Where:** README.md - README.md:208-209 ('5.1 m (95% CI 4.8-6.2), 53% within 5 m across eight clips'); HANDOFF.md:36,63  _(accuracy/cv)_
- **Issue:** The README presents 5.1 m mean and 53%-within-5 m as the validated accuracy, and HANDOFF.md repeats '~5.1 m vs SoccerNet' (line 36) and '~5 m zone-grade' (line 63). As shown in the two findings above, those numbers are computed over a 12 m-gated, matched-only subset with a per-clip GT-selected flip, and the 95% CI is bootstrapped over only 8 per-clip means (validate_topdown.py:178-181) — a thin basis for a 4.8-6.2 band. None of these conditions are stated where the number is quoted. The honest-scope language elsewhere ('visible players only', 'heatmap-grade') is good but does not cover the selection bias in the metric itself.
- **Fix:** Annotate the claim with: matched-within-12 m only, orientation resolved per clip, n=8 clips for the CI. Consider reporting an unconditional metric (detection precision/recall at 5 m) so the headline is not conditioned on already-good matches.

### 14. Spread gate uses a fixed 40-pixel threshold against variable-resolution input
- **Where:** src/homography.py - spread_ok lines 79-85 (np.ptp(pts[:,0]) > 40 and np.ptp(pts[:,1]) > 40)  _(accuracy/cv)_
- **Issue:** spread_ok rejects keypoint sets unless their image-pixel bounding box exceeds 40 px in both x and y. This threshold is absolute pixels, but keypoint_homography/detect explicitly support different imgsz and the docstrings tout raising resolution ('Higher imgsz finds more pitch keypoints on high-res frames', homography.py:90; broadcast_track.py imgsz note). On 480p a 40 px span is ~5% of width (a meaningful spread requirement); on 1080p/4K SoccerNet frames the same 40 px is trivially passed by a tiny, near-collinear cluster, so the gate stops protecting against ill-conditioned homographies exactly on the high-res inputs the code is tuned to prefer. The SVD floor (s[-1] > 1e-3) is also an unnormalised absolute and scales with the same problem.
- **Fix:** Scale the spread requirement to frame dimensions (e.g. ptp > 0.15 * width / height) or to the keypoint set's resolution, so the gate's geometric meaning is resolution-invariant.

### 15. Min-width floor distorts the win-prob bar so it contradicts its own labels
- **Where:** src/theme_broadcast.py - _draw_winprob, lines 199-201 (MINW floor + renormalize); editorial _draw_winprob line 137; telemetry _draw_winprob line 159  _(accuracy/dashboard)_
- **Issue:** All three themes give each segment a minimum width then renormalize. In broadcast: ws=[max(r*bw,MINW) if r>0 else 0 ...]; ws=[w*(bw/sum(ws))...] with MINW=bw*0.045. At a lopsided state (ARG 100.0%, draw ~4e-7, away ~2e-5 at preview minute 64), the rendered widths become ~0.917/0.041/0.041: the ARG segment paints at 92% of the bar while the numeric label says 100%, and two clearly visible ~4% stubs appear for draw/away that the text correctly reports as ~0%. The bar therefore visually contradicts the percentages printed beside it. Editorial (floor 0.012) and telemetry (floor 0.012) have the same defect, less severe. Confirmed numerically against _preview_data at ti=64.
- **Fix:** Only apply the min-width floor to segments above a visibility threshold (e.g. >=1%), or drop the floor and let true-zero segments collapse; render the divider lines without inflating segment widths so the bar geometry matches the displayed percentages.

### 16. Goals after minute 98 dropped from score/xG/win-prob while still counted in final score
- **Where:** src/match_data.py - lines 132-161 (mins = np.arange(0,99); cum loops `for t in mins`); final_h/final_a at 121-127  _(accuracy/dashboard)_
- **Issue:** load() builds every per-minute series over mins = arange(0,99), so any goal/shot with min >= 99 is silently excluded from sc_h/sc_a, xg_h/xg_a, wp_*, and wp_xg. But final_h/final_a (lines 121-127) and the 'correct' badge count ALL goals regardless of minute. Verified: a goal at time=93 (90+3) is captured (index 93<99), but a goal at minute 105 (extra time) is dropped from sc_h (yields 0) while final_h counts it. For a knockout match decided in ET — or any goal FotMob timestamps >=99 — the header score m['sc_h'][98] shown at the playhead end will disagree with final_h, the goal never appears in the xG race or ticker, and the win-prob curve never reflects it. The displayed Argentina 3-0 group game is unaffected, but the layer is wrong for the WC2026 knockout fixtures the project targets.
- **Fix:** Size mins to the actual max event/shot minute (e.g. arange(0, max(99, last_min+1))) or clamp/bucket >=90 minutes into the last index; and assert sc_h[-1]==final_h / sc_a[-1]==final_a so the header score and the 'correct' badge can never diverge.
